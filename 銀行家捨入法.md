# 你的程式四捨五入算錯了嗎？認識銀行家捨入法

身為程式設計師，你一定寫過這樣的程式：

```csharp
Math.Round(3180.5)  // 你以為答案是 3181？
```

執行看看，結果是 **3180**。

這不是 bug，這是 C# 的預設行為——**銀行家捨入法（Banker's Rounding）**。

---

## 什麼是銀行家捨入法？

銀行家捨入法的正式名稱是「**四捨六入五成雙**」，英文叫 **Round Half to Even**。

規則很簡單：當要進位的那一位數字**恰好是 5** 時（前後位都沒有其他數字影響），不是無條件進位，而是根據**前一位數字的奇偶性**決定方向：

- 前一位是**奇數** → **進位**（讓結果變成偶數）
- 前一位是**偶數** → **捨去**（結果保持偶數）
- 不是 5 → 跟傳統四捨五入一樣

換一句話說：**永遠朝最近的偶數靠攏**。

### 看幾個例子（進位到整數）

| 原始值 | 銀行家捨入 | 傳統四捨五入 | 差異 |
|--------|-----------|--------------|------|
| 0.5    | **0**（0 是偶數） | 1  | 不同 |
| 1.5    | **2**（2 是偶數） | 2  | 相同 |
| 2.5    | **2**（2 是偶數） | 3  | 不同 |
| 3.5    | **4**（4 是偶數） | 4  | 相同 |
| 4.5    | **4**（4 是偶數） | 5  | 不同 |
| 5.5    | **6**（6 是偶數） | 6  | 相同 |
| 3180.5 | **3180**          | 3181 | 不同 |
| 3181.5 | **3182**          | 3182 | 相同 |

規律很明顯：當 .5 前面是偶數，就捨去；前面是奇數，就進位。

---

## 為什麼要這樣設計？

傳統四捨五入有一個統計上的缺陷：**永遠把 .5 往上進**。

想像一個場景：你有一百萬筆利息計算，每筆金額的小數部分剛好都是 .5。用傳統四捨五入，每一筆都進位，加總後的總金額會**系統性偏高**，長期下來誤差越積越大。

銀行家捨入法的設計思想是：在大量計算中，.5 遇到偶數捨去、遇到奇數進位，**進位與捨去的機率各約 50%**，加總後誤差趨近於零，對統計更公平。

這個方法原本廣泛用於銀行業的大量利息計算，因而得名「銀行家捨入法」。它同時也是 **IEEE 754 浮點數標準**指定的預設捨入方式，也就是說，大多數 CPU 的浮點運算就是這樣捨入的。

---

## 哪些語言預設採用銀行家捨入法？

這裡列出常見語言與工具的預設行為，結果可能跟你想的不一樣：

| 語言 / 工具 | 預設捨入方式 | 備註 |
|-------------|-------------|------|
| **C#** `Math.Round(x, n)` | **銀行家捨入** | 需加參數才能改為傳統四捨五入 |
| **VB.NET** `Math.Round(x, n)` | **銀行家捨入** | 同 C# |
| **RDLC / SSRS** `Round(x, n)` | **銀行家捨入** | 使用 VB.NET 運算式引擎 |
| **Python 3** `round(x, n)` | **銀行家捨入** | Python 2 以前是傳統四捨五入 |
| **IEEE 754 浮點數標準** | **銀行家捨入** | 大多數 CPU 預設行為 |
| **Java** `Math.round(x)` | 傳統四捨五入 | 等同 `floor(x + 0.5)` |
| **JavaScript** `Math.round(x)` | 傳統四捨五入 | |
| **SQL** `ROUND(x, n)` | 傳統四捨五入 | 大多數 DBMS |
| **Excel** | 傳統四捨五入 | |
| **計算機** | 傳統四捨五入 | |

**一個容易踩到的坑**：C#、Python、RDLC 預設是銀行家捨入法，但 Excel 和計算機是傳統四捨五入。如果系統用 C# 計算，使用者拿 Excel 驗算，**當數值的小數恰好是 .5 時，兩邊結果不同**，使用者會以為系統算錯了。

## 實際踩坑案例

這是一個真實發生的財務系統問題：

```
匯率：31.805
金額：100 元（外幣）
精確值：31.805 × 100 = 3180.5 元（台幣）
```

畫面顯示：**3,180**
使用者用計算機算：**3,181**

使用者回報 bug，但系統其實沒有寫錯——只是 C# 的 `Math.Round()` 預設用了銀行家捨入法，而使用者的計算機用的是傳統四捨五入。

---

## 如何在 C# 指定傳統四捨五入？

`Math.Round()` 有一個額外參數 `MidpointRounding`，可以明確指定捨入方式：

```csharp
// 銀行家捨入（預設）
Math.Round(3180.5);                                           // 3180
Math.Round(3180.5, MidpointRounding.ToEven);                  // 3180

// 傳統四捨五入
Math.Round(3180.5, MidpointRounding.AwayFromZero);            // 3181

// 保留 2 位小數
Math.Round(3180.55, 2, MidpointRounding.AwayFromZero);        // 3180.55（視值而定）
```

財務系統建議的寫法：

```csharp
// 逐列計算
amount += Math.Round(exchangeRate * expense, 2, MidpointRounding.AwayFromZero);

// 最終取整數
int total = (int)Math.Round(subtotal, 0, MidpointRounding.AwayFromZero);
```

---

## Python 也一樣

```python
round(0.5)   # 0（銀行家捨入！）
round(1.5)   # 2
round(2.5)   # 2（銀行家捨入！）
round(3.5)   # 4

# 想要傳統四捨五入，用 decimal 模組
from decimal import Decimal, ROUND_HALF_UP
Decimal('2.5').quantize(Decimal('1'), rounding=ROUND_HALF_UP)  # 3
```

---

## RDLC 報表也有同樣問題

RDLC / SSRS 的 `Round()` 函式使用 VB.NET 運算式引擎，同樣預設銀行家捨入法：

```vb
' 銀行家捨入（預設）
Round(3180.5, 0)   ' → 3180

' 傳統四捨五入（需使用 Math 類別）
Math.Round(3180.5, 0, MidpointRounding.AwayFromZero)   ' → 3181
```

## 總結

| | 銀行家捨入法 | 傳統四捨五入 |
|---|---|---|
| **又稱** | Round Half to Even、四捨六入五成雙 | Round Half Away from Zero |
| **規則** | .5 朝最近偶數捨入 | .5 一律進位 |
| **統計特性** | 大量計算後誤差趨近於零 | 大量 .5 會系統性偏高 |
| **適合場景** | 科學計算、統計、金融後台 | 日常使用、對帳、使用者驗算 |
| **C# 預設** | ✅ 是 | — |
| **Excel 預設** | — | ✅ 是 |

**給開發者的建議：**
1. 寫財務相關計算，永遠明確指定 `MidpointRounding.AwayFromZero`，不要依賴預設值
2. 使用者會用 Excel 或計算機驗算，系統的捨入行為應與他們一致
3. 團隊內有跨語言（C# + SQL + Excel）的計算流程時，捨入行為要統一確認

銀行家捨入法不是 bug，它在統計上其實更嚴謹——只是對一般使用者來說，不符合從小學到的「五就進一」直覺。知道這個差異，下次遇到「系統算錯了」的回報，或許就能快速定位原因。
